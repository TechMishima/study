# TIL マネジメント

## 1.プロジェクトマネジメントを理解する

### プロジェクトの定義

決められた期限内で、特定の目的を達成する活動

### フィージビリティスタディ

新しいプロジェクトなどの計画に対して、その実行可能性を評価するために調査・検証をすること

### PMBOK(Project Management Body Of Knowledge)

プロジェクトマネジメントに関する知識を体系的にまとめたもの(プロジェクトマネジメントの世界標準)

- 5つのプロセス
  - 立ち上げ
  - 計画(Plan)
  - 実行(Do)
  - 監視・コントロール(Check,Action)
  - 終結

- 10個の知識エリア
  - 統合管理
    - 他の9つの知識エリアを取りまとめて、全体をマネジメントする
  - スコープ管理
    - プロジェクトの作業範囲を決め、成果物とタスクを定義する
    - WBS(Work Breakdown Structure) : 作業分解構造図<br>
    プロジェクトで行う作業を階層的に分解する
  - スケジュール管理
    - 納期を守るために、作業スケジュールや日程を細かく管理する
    - ガントチャート : プロジェクトの進捗を管理するためのスケジュール表
    - アローダイアグラム : 作業の順序や所要時間などの情報を組み合わせて、プロジェクト全体の流れを図示したもの
  - コスト管理
    - プロジェクトにかかる費用を見積り、予算内におさめるよう管理する
      - 「ファンクションポイント法」 : 画面数・帳票数・ファイル数などを基準に、機能の難易度を定義して見積もる
      - 類推見積法 : 過去の類似プロジェクトの実績値を参考に見積もる
      - プログラムステップ法 : LOC法とも呼ばれ、プログラムのステップ数を基に見積もる
      - COCOMO法 : LOC法を基に、開発スキルや難易度などを考慮して見積もる
  - 品質管理
    - プロジェクトの成果物の品質を管理する
  - 資源管理
    - プロジェクトに必要な人材、物的資源を管理する
  - コミュニケーション管理
    - チームメンバー、ステークホルダーとの情報交換を管理する
    - ステークホルダー : 株主・経営者・顧客・取引先などの利害関係者
  - リスク管理
    - プロジェクトで発生する可能性があるリスクを管理する
  - 調達管理
    - プロジェクトに必要なサービスや製品の調達や契約を管理する
  - ステークホルダ管理
    - ステークホルダーとの関係を管理する

## 2.コスト管理とスケジュール管理を理解する

### ファンクションポイント法

機能の数と各機能の難易度を基に開発コストを見積もる

例）
```
機能1 : 機能の数1個 × 難易度 簡単 10点 = ﾌｧﾝｸｼｮﾝﾎﾟｲﾝﾄ 10点
機能2 : 機能の数3個 × 難易度 簡単 10点 = ﾌｧﾝｸｼｮﾝﾎﾟｲﾝﾄ 30点
機能3 : 機能の数2個 × 難易度 普通 30点 = ﾌｧﾝｸｼｮﾝﾎﾟｲﾝﾄ 60点
機能4 : 機能の数1個 × 難易度 難しい 50点 = ﾌｧﾝｸｼｮﾝﾎﾟｲﾝﾄ 50点

合計 150点
```

### 開発工数

工数とは、ある作業を完了するために必要な時間

「人月」 : 1人が1か月でこなすことができる作業量

### アローダイアグラム

テスト出題ポイント
- プロジェクト全体の作業日数
- クリティカルパス
- 最早結合点時刻
  - 日数の大きいほうが最早結合
- 最遅結合点時刻
  - 日数の小さいほうが最遅結合

## 3. ITサービスマネジメントとは

### ITサービスマネジメント

利用者が必要とするITサービスを継続的に提供またはサポートできるように管理すること

「ITIL」(Information Technology Infrastructure Library)<br>
ITサービスマネジメントに関する成功事例を体系的にまとめたもの

### ITサービス運用方法

ITサービスマネジメントは、「日々のサービス運用」と「中長期的なサービス運用」がある

### サービスデスク

ITサービス利用者とITサービス提供者の間の単一窓口<br>
トラブル時の対処方法、使用方法、苦情など、さまざまな問い合わせを受け付け

- 中央サービスデスク
  - サービスデスクを一か所に集中
- ローカルサービスデスク
  - 利用者の近くにサービスデスクを配置
- バーチャルサービスデスク
  - 各地にスタッフが分散していてもITツールで仮想的に一か所に集中

### 問い合わせ対応とインシデント管理

インシデント管理及びサービス要求管理によってなりたつ

１）要求管理
```
クライアント
 ↓ サービス要求
要求管理
```

２）インシデント問い合わせ
```
クライアント
 ↓ インシデント問合せ
インシデント管理
 ↓ 
記録
 ↓ 
優先度の割り当て・分類
 ↓ 
調査と診断 「既知のエラーDB」
 ↓ 解決できた場合
インシデントクローズ
```

３）インシデント問い合わせ ※デスクで対応不可
```
クライアント
 ↓ インシデント問合せ
インシデント管理
 ↓ 
記録
 ↓ 
優先度の割り当て・分類
 ↓ 
調査と診断 「既知のエラーDB」

 ↓ 解決できなかった！

段階的取り扱い(エスカレーション)
 ↓ 
問題管理
 ↓ 
変更管理 : 変更に伴う影響を検証、承認または却下
構成管理 : ITサービスの提供に必要な情報を正しく把握する 「構成管理DB」
 ↓ 
リリース管理 : 承認された変更内容を、適切な時期にリリースする
```

### 中長期的なサービス運用

- サービスレベル管理
  - 利用者の求めるサービスレベルを満たしているか評価・管理
  - 「SLA」Service Level Agreement : サービス品質保証/合意書<br>ITサービスの提供者とITサービスの利用者の間で、どの程度のサービス品質を保証するか決めたもの
- 可用性管理
  - サービスを利用したい時に確実に利用できるように、個々の機能を維持管理
- キャパシティ管理
  - ITサービスの供給に必要なネットワークやシステムなどの容量・能力を管理

## 4.システム監査を完璧に

### システム監査の概要

監査対象からは独立した立場(第三者の立場)となる、システム監査人が行う<br>
情報システムを信頼性、安全性、効率性などの観点で評価し、問題点を指摘したり改善策を助言する

### システム監査の登場人物

- 監査対象部門(被監査部門)
- 監査依頼人(経営者)
- システム監査人

監査をする人は「独立性」を担保することが重要
- 外観上の独立性
  - 監査対象と利害関係にならないように、「独立した部署・組織」でなくてはならない
- 精神上の独立性
  - 監査の実施にあたり、客観的な視点から「公正な判断」を行わなければならない

### システム監査の流れ

```
1. 監査計画の作成
依頼人の意向や経営課題を調査して目的・対象を明らかにしたうえで計画を立案する

2. 監査実施
監査計画を基に予備調査・本調査を行い、「入手した監査証拠に基づいて」評価する

3. 監査報告
監査報告書をもとに、遅延なく監査依頼者に結果を報告する

4. フォローアップ
監査対象部門による改善の実施状況をモニタリングし、システム監査人の立場で「改善の実現を支援」する
改善命令をするのは監査依頼人なので、「システム監査人は、あくまで指導と助言のみ」
```

### コーポレートガバナンスと内部統制

企業が不祥事や法令違反などを起こさず、健全な経営を行うために作られたルール・仕組み<br>
上場会社においては、内部統制の整備・運用が義務付けられている

- コーポレートガバナンス(企業統治)
  - 経営者が不正を行わないように、株主や監査役が企業経営そのものを監督・監視する仕組み
- 内部統制
  - 従業員が不正を行わないように、経営者が企業内を管理する仕組み
