# TIL システム開発

## 1.システム開発の流れ

### システム開発の流れ

企業がシステムを開発する際、通常はSier等のシステム開発会社に依頼する。<br>
開発依頼する側を「ユーザー企業」、依頼を受ける側が「ベンダ企業」と呼ばれる。<br>
お互いの認識をすり合わせるために、「共通フレーム」を使用する。

<b>共通フレーム</b><br>
ソフトウェア開発の工程や作業内容・作業範囲を定義したガイドライン

- 企画プロセス
  - ニーズや課題を確認「システム化構想」
  - 具体的な計画を立てる「システム化計画」
- 要件定義プロセス
  - 詳細機能、仕様を明確にする
- 開発プロセス
  - 要件をもとに設計、開発する
- 運用プロセス
  - システムを運用する
- 保守プロセス
  - 問題点の修正などメンテナンスを行う

### RFIとRFP

ユーザー企業がどうやってベンダ企業を見極めるか

- 「RFI」(Request For Information) 情報提供依頼書
  - 一般的な情報の提供を求める(一次審査)
- 「RFP」(Request For Proposal) 提案依頼書
  - 個別の具体的な提供を求める(二次審査)

### ソフトウェア開発工程

要件定義と開発のプロセスの細分化

「V字モデル」(設計工程とテスト工程が紐づいている)
- 要件定義
  - 必要とする機能や要求を定義
  - 「機能要件」 : ユーザーが求めている機能
  - 「非機能要件」 : 機能以外の要件(プログラミング言語の選定など)
- 外部設計(基本設計)
  - システム利用者に影響する部分を設計
- 内部設計(詳細設計)
  - システム開発に影響する部分を設計
- コーディング
- 単体・結合テスト
  - プログラムを構成する機能単位でテストする
- 総合テスト(システムテスト)
  - 本番運用を想定して、システム全体の動作検証を行う
- 受入テスト(UAT)
  - ユーザーの要求どおりに完成しているか検証する

### レビュー

要件定義、外部設計、内部設計の工程の終わりに「レビュー」を実施して設計不備や誤りを早期に発見する

- ウォークスルー
  - 担当者が関係者に対して成果物の内容を説明する
- インスペクション
  - モデレーターが司会進行役となり、参加者の役割を明確にして進める
- ラウンドロビン
  - 参加者全員がテーマごとに持ち回りで責任者となり進める

### EA(エンタープライズアーキテクチャ)

「EA」 : システム開発で重要な考え方<br>
一つ一つのシステムを最適化するのではなく、企業全体で見たときの最適化を目指す

- ビジネスアーキテクチャ
  - 企業の戦略実現に必要な業務内容、業務プロセスなど
- データアーキテクチャ
  - 業務を行う上で必要なデータ、データの関連性
- アプリケーションアーキテクチャ
  - データを処理する情報システムの機能
- テクノロジーアーキテクチャ
  - システムの構築に必要な技術的構成要素

## 2.システム開発手法（アジャイル・ウォーターフォール等）

### システム開発の手法の一覧

- ウォーターフォール
  - 「上流から下流へ順番」に工程を進める
- アジャイル
  - 「短い開発サイクル」を回す
- プロトタイピングモデル
  - 「試作品を作成」し、利用者がチェックを行いながら進める
- リバースエンジニアリング
  - 「開発工程をさかのぼり」、設計書を取り出す

### ウォーターフォール

要件定義・設計・開発・テストなどの工程を、上流工程から下流工程へ順番に進めていく手法

- メリット
  - 全体スケジュールが立てやすく、計画通りに開発を進めやすい
- デメリット
  - 途中の仕様変更を取り入れにくい
  - 不具合が発生すると、後戻りが難しい

### アジャイル

小さな機能単位ごとに開発サイクルを回し、機能ごとにリリースし段階的にシステム全体を完成させる

- メリット
  - 開発期間が短い
  - 仕様変更に柔軟に対応できる
- デメリット
  - 全体スケジュールが立てにくく、進捗が把握しづらい
  - 開発の方向性がずれやすい

### アジャイル開発(エクストリームプログラミング、XP)

<font color="magenta">
よく出題される！
</font>

アジャイル開発手法の一つで、顧客の要望を取り入れながら柔軟に開発を進める方法<br>
設計・開発・テスト・改善のサイクル(=イテレーション)を繰り返して開発する

5つの価値と19のプラクティスによって構成されている

- 「ペアプログラミング」
  - コードを書く人とコードをチェックする人の役割に分担して、「2人1組」でプログラミングを行う
- 「リファクタリング」
  - 外部からの挙動は変えずに、「プログラムの内部構造を変更」する
- 「テスト駆動開発」
  - プログラム実装前に「先にテストコードを書いて」、そのテストを通るようにコードを書く

### アジャイル開発(スクラム)

アジャイル開発手法の一つで、開発チームが一体となり短期間で開発を進める<br>
スプリント(1~4週間)の短い期間を繰り返す。またデイリースクラム(毎日)を行い進捗や課題の確認を行う。

### プロトタイピングモデル

開発の早い段階で試作品(プロトタイプ)を作り、利用者の確認をとりながら開発を進めていく手法

- メリット
  - 利用者との認識のズレが発生する可能性が低く、手戻りになりにくい
- デメリット
  - プロトタイプを作るのにコストがかかる

### リバースエンジニアリング

既存の製品・プログラムを解析して、その仕組みや使用、設計を明らかにする開発手法

- メリット
  - 開発にかかるコストと時間を削減できる
- デメリット
  - 特許権や著作権の侵害に当たる可能性がある

## 3.業務モデリングとユーザーインターフェース

### 業務モデリング手法

要件定義を行う前にユーザー業務のヒアリングを行う<br>
ヒアリングした業務プロセス等を業務モデルとして図式化する

- 「E-R図」
  - データベース設計で使用され、実体(対象業務を構成するもの)と実体の関連をモデル化したもの
- DFD
  - システムにおけるデータの流れをモデル化したもの
- UML
  - オブジェクト指向のシステム開発で使用され、分析から設計、実装まで一貫した表記方法でモデル化したもの

### E-R図

データベース設計で使用され、実体(対象業務を構成するもの)と実体の関連をモデル化したもの

- 実体 = エンティティ(Entity) ※テーブル
- 関連 = リレーションシップ(Relationship)

例）
```
多 体 1
A ← B

BはAを複数持てるが、AはBを1つしか持てない
```

### DFD(Data Flow Diagram)

システムにおけるデータの流れをモデル化したもの

例）
```
DFDの記法

→ データフロー : データの流れ
〇 プロセス : データの処理
= データストア : ファイル、データベースなどのデータ格納場所
□ データの源泉と吸収 : データの始まりと終わり
```

### UML

基本情報技術者試験9.mdの5章に説明あり。

### UI ユーザーインターフェース

ユーザーとコンピューターの接点となるもの。<br>
ユーザーとコンピューター間で情報をやり取りする機器や入力装置

- 画面設計
  - 画面の項目やレイアウトなど、データを入力する際の操作方法を設計
- 帳票設計
  - 帳簿や伝票などのレイアウト、プリンタに出力する内容などを設計

### GUIとCUI

- GUI(グラフィカル ユーザー インターフェース)
  - 画面上のアイコンやボタンなどマウスで操作する視覚的なインターフェース
- CUI(キャラクター ユーザー インターフェース)
  - キーボードによる文字入力を用いたインターフェース(コマンドプロンプトなど)

### 過去問演習

- チェックボックス
  - いくつかの項目について、それぞれの項目を選択するかどうかを指定する。
- コンボボックス
  - 幾つかの選択項目から一つを選ぶときに、選択項目にないものはテキストボックスに入力する。
- ラジオボタン
  - 互いに排他的ないくつかの選択項目から一つを選ぶ。
- プルダウンメニュー
  - 特定の項目を選択することによって表示される一覧形式の項目から一つを選ぶ。

## 4.モジュール分割を理解する

### モジュールの説明

モジュール : システムや機器を構成する1つの機能や要素のこと

### モジュール分割

システムをモジュール単位で開発・テストを行っていく

- メリット
  - 複数人で並行して作業を進めることができる
  - 共通する機能を他のところでも再利用できる
  - プログラムに問題が起こった場合、一部の修正でよい

### モジュールの独立性

他のモジュールとの関係度合いは少ない「独立性」が高いと良い。<br>
モジュールの変更が起きたとき、他のモジュールへの影響が少なくなる。

モジュールの独立性を表す指標として、「モジュール強度」と「モジュール結合度」がある

### モジュール結合度

モジュール同士の関連性の強さを表す<br>
モジュール結合度が弱いほうが、モジュールの独立性が高くなり、システムメンテナンスがしやすい

上のあるほどモジュール結合度が弱い
- 「データ結合」
  - 処理に必要なデータだけを引数をして渡す
- スタンプ結合
  - データ構造(データのまとまり)を引数として渡す
- 制御結合
  - モジュールの実行順序を制御するデータ(制御パラメータ)を引数として渡す
- 外部結合
  - 外部宣言されたデータを参照する
- 共通結合
  - 外部宣言されたデータ構造(データのまとまり)を参照する
- 内容結合
  - 他のモジュール内のデータを参照する

### モジュール強度

1つのモジュールが1つの機能に洗練(特化)されているか<br>
モジュール強度が強いほうが、モジュールの独立性が高くなり、システムメンテナンスがしやすい

上にあるほどモジュール強度が強い
- 「機能的強度」
  - 一つの機能のみを持つモジュール
- 情報的強度
  - 同じデータを使う複数の機能をまとめたモジュール
- 連絡的強度
  - データの受け渡し・参照をしながら、順番に実行される複数の機能をまとめたモジュール
- 手順的強度
  - 順番に実行される複数の機能をまとめたモジュール
- 時間的強度
  - ある特定の時点(プログラム開始時など)で実行する機能をまとめたモジュール
- 論理的強度
  - 関連する複数の機能をまとめたモジュール
- 暗号的強度
  - 関係のない機能をまとめたモジュール

### テストとバグの関係

テストを行う目的は、プログラムのバグを発見し修正すること<br>
「信頼度成長曲線」や「バグ管理図」を見ることで、テストの進捗や品質を確認する

### 単体テスト(ブラックボックステスト)

モジュールごとにテストを行う、単体テストの一種<br>
内部構造は気にせず、入出力の結果のみを確認する。<br>
テストケースは「限界値分析・同値分析」の考え方で作成する。

### 単体テスト(ホワイトボックステスト)

プログラムの内部構造に着目してテストを行う。<br>
「命令網羅・判定条件網羅・条件網羅・複数条件網羅」でパターンを確認する

### 結合テスト

単体テストが完了したモジュール同士を結合して行う

- 「トップダウンテスト」
  - 上から下に順番に結合
  - もし下位モジュールが完成していなかったら、「スタブ」という仮の下位モジュールを使用する
- 「ボトムアップテスト」
  - 下から上に順番に結合
  - もし上位モジュールが完成していなかったら、「ドライバ」という仮の上位モジュールを使用する

### 総合テスト

本番とほぼ同じ環境で、「開発者が」システム全体の動作検証を行う

暗記不要
- 機能テスト
  - 必要な機能が全て含まれており、要件通りに動作するか確認する
- 性能テスト
  - システムの処理速度などパフォーマンスを確認する
- 負荷テスト
  - システムに大きな負荷をかけ、負荷がかかった状態でも正常に動作するか確認する
- セキュリティテスト
  - システムが攻撃されても、要件通りセキュリティを保持できるかを確認する
- ユーザビリティテスト
  - 操作がしやすいか、画面が見やすいかなどユーザーにとって使いやすいかどうかを確認する

### 受入テスト

本番とほぼ同じ環境で、ユーザーの要求通りに完成しているか「ユーザーが」テストを行う

### リグレッションテスト(回帰テスト)

システムの修正や変更によって、別の部分に不具合が発生していないかをチェックする
